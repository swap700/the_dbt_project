# dbt-powered E-Commerce Analytics Pipeline

> **Brazilian E-Commerce (Olist) · dbt Core · Snowflake · Tableau**

A production-quality dbt project demonstrating a full Raw → Staging → Intermediate → Marts architecture on the beloved [Olist Brazilian E-Commerce dataset](https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce) — 100k+ orders, 8 relational tables, real-world messiness.

---

## The Problem This Solves

> *"Our SQL transforms are scattered across 47 ad-hoc scripts, nobody knows which one is the source of truth, and every time someone changes a table upstream, three dashboards break silently."*

This is the Analytics Engineering problem. dbt solves it by:

| Without dbt | With dbt |
|---|---|
| SQL scripts in random folders | Versioned models in a defined layer structure |
| No tests — silent failures | Automated tests on every model, every run |
| No lineage — mystery dependencies | Visual DAG showing every upstream/downstream relationship |
| Copy-paste logic across scripts | `ref()` — reusable, DRY model references |
| "Works on my machine" | Profiles + CI = reproducible anywhere |

## Architecture: Why Three Layers?

### Staging (`models/staging/`)
**One model per source table. Views. No business logic.**

Staging models do exactly four things:
1. Rename columns to a consistent convention
2. Cast data types (raw Snowflake TEXT → proper TIMESTAMP, FLOAT, INTEGER)
3. Handle nulls at the source
4. Add cheap derived columns (e.g., `delivery_delay_days = estimated - actual`)

Why views? They're always fresh, cost nothing to store, and are cheap to recompute. If a raw table changes, the view reflects it instantly.

### Intermediate (`models/intermediate/`)
**Multi-table joins and aggregations. Ephemeral. Business logic lives here.**

Intermediate models are `ephemeral` — no database object is created. They get compiled as CTEs inside the downstream mart models. This means:
- Zero query overhead (no extra table scan)
- Reusable: `int_orders__joined` is used by *both* `fct_orders` and `fct_revenue_daily`
- One place to change logic that affects multiple marts

Three intermediate models:
- `int_orders__joined` — enriches every order with items, payments, and reviews
- `int_customer_orders__grouped` — aggregates orders to the TRUE customer level (CLV)
- `int_seller_metrics__grouped` — computes seller performance scores

### Marts (`models/marts/`)
**Final tables that Tableau queries. Pre-aggregated. Fast.**

| Model | Grain | Powers |
|---|---|---|
| `fct_orders` | One row per order | Order-level drill-down |
| `fct_revenue_daily` | One row per day | Revenue trend charts |
| `dim_customers` | One row per unique customer | CLV, segmentation |
| `dim_sellers` | One row per seller | Seller performance dashboard |

## Data Quality: dbt Tests

Every model has tests defined in `schema.yml` files:

```yaml
# Example from _core__models.yml
- name: fct_orders
  columns:
    - name: order_id
      data_tests:
        - not_null
        - unique
    - name: order_status
      data_tests:
        - accepted_values:
            values: [delivered, shipped, canceled, invoiced, ...]
    - name: customer_id
      data_tests:
        - relationships:
            to: ref('stg_olist__customers')
            field: customer_id
```

Run all tests: `dbt test`  
Run tests on one model: `dbt test --select fct_orders`

This is what separates someone who *uses* dbt from someone who *understands data quality* — tests make it impossible for a dashboard to silently break when an upstream table changes.

## Setup Guide

### 1. Get the Data
Download from Kaggle: https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce  
Place all 9 CSV files in this project directory.

### 2. Load to Snowflake
```bash
pip install snowflake-connector-python
# Edit credentials in load_data_to_snowflake.py, then:
python load_data_to_snowflake.py
```

### 3. Configure dbt Profile
```bash
# Copy profiles.yml to ~/.dbt/profiles.yml
cp profiles.yml ~/.dbt/profiles.yml
# Edit ~/.dbt/profiles.yml with your Snowflake credentials
```

### 4. Install & Run dbt
```bash
pip install dbt-snowflake
dbt deps          # install dbt-utils package
dbt run           # build all models
dbt test          # run all data quality tests
```

### 5. Generate & View Docs
```bash
dbt docs generate
dbt docs serve    # opens http://localhost:8080
# Screenshot the lineage graph from the Lineage tab
```

### 6. Connect Tableau
In Tableau Desktop:
- **Connector**: Snowflake
- **Server**: your account identifier
- **Database**: `OLIST_DB`
- **Schema**: `DEV_MARTS` (or `PROD_MARTS`)
- **Tables to connect**: `FCT_ORDERS`, `FCT_REVENUE_DAILY`, `DIM_CUSTOMERS`, `DIM_SELLERS`

## Key Modeling Decisions

**Why is `customer_id` not the customer PK?**  
Olist creates a new `customer_id` for every order. The same person buying three times has three `customer_id`s. `customer_unique_id` is the stable identity — this is a common raw data quirk that staging models are designed to catch and clarify.

**Why is `int_orders__joined` ephemeral?**  
It's referenced by both `fct_orders` and `fct_revenue_daily`. Making it ephemeral means its logic is inlined into both (no extra table scan), while keeping the code DRY. Change the join once, both marts reflect it.

**Why pre-aggregate to `fct_revenue_daily`?**  
Tableau querying 100k rows on every dashboard load is slow. A pre-aggregated daily grain table has ~730 rows (2 years of data) — instant.

**Why are staging models views?**  
Staging is a thin transformation layer on raw data. Views are always fresh, storage-free, and fast to recompute. Tables are reserved for marts where query speed matters.
